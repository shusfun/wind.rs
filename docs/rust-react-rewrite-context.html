管理台认证

检查初始化

GET /setup/status 返回是否需要初始化。初始化完成后禁止再次 setup。

首次设置 key

用户在 /setup 录入或生成管理 key，后端保存 key hash，并写入安装锁。

登录和会话

/login 输入 key，后端签发本地管理 session token。管理 API 默认使用 session token。

证据要求

request_traces 必须能定位“输出一直不停”的原因。每条请求至少记录请求 ID、模型、stream、选中账号、开始时间、结束时间、结束原因、错误摘要。

流式证据需要分层记录：客户端原始请求、远端原始事件、转换后下游事件。

批量登录要求

执行方式

批量任务严格串行，每个账号之间随机延迟，支持停止剩余任务。

前端进度

实时显示当前账号、总进度、等待倒计时、成功数、失败数和失败原因。

默认延迟

成功或普通失败后 15-45 秒；风控或限频类失败后 60-180 秒；可在 settings 中调整。

调度器要求

账号状态

  • ready：可被选择。
  • cooldown：临时冷却。
  • blocked：等待人工或后台恢复。
  • disabled：禁用。

调度行为

  • 按模型过滤可用账号。
  • 优先级高的账号先参与选择。
  • 同优先级轮询。
  • 每次请求拿到租约，结束后释放。
  • 失败通过 result reporter 更新账号状态。
  • 同一请求内失败账号加入排除集。

错误冷却建议

  • 401 / 403：长冷却。
  • 429 / 风控：按 retry-after 或指数退避。
  • 5xx / LS 异常：短冷却。
  • 账号凭据失效：blocked 或 disabled，等待人工处理。

Claude Code / Anthropic 首版要求

  • 支持 stream 与非 stream。
  • 支持 Claude Code 常见请求体字段。

  • 错误响应尽量使用 Anthropic 风格。

重点验证

  • Claude Code Opus 请求能正常结束。
  • 工具调用相关内容不导致无限流。
  • sub2api 接入后不会出现长时间 token 下行不停。
  • 上游 LS 异常时能返回明确错误，而不是悬挂连接。
