CPA 调度观察

核心结构

CPA 的 authScheduler 按 provider 拆分，再按 model 建 shard。每个 shard 内部保存 ready 队列和 cooldown 队列。

  • providerScheduler：一个服务商的账号视图。
  • modelScheduler：一个模型下的账号调度视图。
  • scheduledAuth：账号在某个模型视角下的运行状态。
  • readyBucket：按优先级组织可用账号。

选择策略

CPA 同时支持 fill-first 和 round-robin。默认思路是先看最高优先级，再在同优先级内轮询。混合 provider 时，会按可用账号数量做加权轮转。

  • 优先级高的账号优先。
  • 同优先级账号避免单点长期占用。
  • 排除本次已尝试账号，避免一个请求内重复踩同一个坏账号。
  • 可为 websocket、会话粘性等路径做选择偏好。
状态 含义 Rust 后端建议
ready 账号可被选择。 只有 ready 账号能获得请求租约。
cooldown 账号临时冷却，等待 nextRetryAt。 429、风控、登录失败、LS 异常都应进入有原因的冷却。
blocked 账号当前不可用，但未必是手动禁用。 用于 refresh 失败、LS 未就绪、代理不可用等中间态。
disabled 账号被人工或策略禁用。 人工禁用与自动禁用要区分原因，方便恢复。

CPA 失败反馈

CPA 的请求执行会把结果反馈给 MarkResult。成功会清理模型级错误；失败会根据状态码设置账号或模型冷却。这个模式值得直接吸收，但退避时长要结合 Windsurf 登录和 LS 行为重新定。 

request -> scheduler.pick -> lease/account
        -> upstream execution
        -> result reporter
        -> success: clear model/account error
        -> failure: set cooldown / quota / blocked reason

普通失败

408、500、502、503、504 走短冷却，CPA 使用约 1 分钟。

认证或权限失败

401、402、403 走较长冷却，CPA 使用约 30 分钟。

限流失败

429 支持 Retry-After，否则指数退避，CPA 基础 1 秒，最高 30 分钟。

sub2api 对接观察

关键约束：如果 sub2api 不改，新 Rust 后端对它暴露的接口必须像一个 OpenAI-compatible upstream。重点不是兼容旧 WindsurfAPI，而是让 sub2api 的现有上游直连路径自然工作。

入站路径

sub2api 在 OpenAI 平台分组下会处理 /v1/chat/completions 和无 /v1 前缀的 /chat/completions。对 APIKey 上游直连时，它会拼出上游 /v1/chat/completions。

  • GET /v1/models：模型列表。
  • POST /v1/chat/completions：主调用路径。
  • POST /v1/responses：后续可选，第一阶段不必强依赖。

流式读取

sub2api 用 scanner 按行读取上游 SSE。只要上游不断开，sub2api 就会继续读。因此新后端必须明确输出结束条件。

  • 响应头：Content-Type: text/event-stream。
  • 每个事件按 SSE 格式输出。
  • 结束时发送 data: [DONE]。
  • 发送完成后主动结束响应。

Rust 新后端最小接口

接口 优先级 行为要求
GET /health P0 返回服务、账号池、LS 管理器的健康摘要。
GET /v1/models P0 返回 OpenAI 兼容模型列表，至少包含 sub2api 会配置的模型名。
POST /v1/chat/completions P0 支持非流式和流式。流式必须有明确结束标记和连接关闭。
POST /admin/login-jobs P1 创建批量 Windsurf 登录任务，严格串行，带随机延迟和失败退避。
GET /admin/login-jobs/:id/events P1 SSE 推送批量登录进度、等待倒计时、失败原因和最终统计。
GET /admin/requests/:id P1 查看原始输入、远端原始输出、转换后输出和结束原因。

建议模块

config

读取本地配置、环境变量、端口、日志路径、账号数据路径、代理配置。

account

账号实体、状态、凭据、代理绑定、冷却、失败计数、订阅/额度快照。

scheduler

参考 CPA 的分片调度，但加入 Rust 的租约模型：拿到账号即占用，drop 或显式 release 释放。

login_job

批量 Windsurf 登录任务系统，支持串行、随机延迟、失败退避、暂停/取消、进度 SSE。

remote_engine

管理远端模板、请求 patch、Windsurf 远端流式调用、超时和错误归类。

openai_compat

处理 OpenAI chat completions 请求、模型映射、SSE 输出、错误格式和 usage 字段。

trace

记录请求证据：原始入参、选中账号、远端原始返回、转换结果、结束原因、耗时和错误。

当前风险

流不结束

之前已经出现“输出一直停不下来”。新后端必须把上游结束、解析结束、客户端断开、超时四类结束原因分开记录。

登录风控

批量登录不能一口气发请求。需要严格串行、随机延迟、失败退避，并把每次请求时间和代理记录下来。

错误归因

需要区分 sub2api、Rust 后端、LS、Windsurf 上游四层错误，否则后面无法判断应该提 issue 还是自己修。
