一句话先懂

旧路线像是在用户电脑上先启动一个“浏览器内核”，再让它帮我们发消息；ZephyrSail 像是已经知道真正要发给后端的请求格式，所以直接组包发给 Windsurf 官方服务器。

结论：如果目标是 4h4g 机器上承载两三百用户、给 sub2api 当上游，ZephyrSail 这种“远端模板调用”路线更轻、更像网关；本地运行时路线更重，更像模拟一个完整 Windsurf 客户端运行环境。

Frontend Analogy

用前端类比理解

旧路线：启动一个隐藏页面再操作它

想象你要获取一个接口结果，但你没有直接调用接口，而是：

  • 先打开一个隐藏 iframe。
  • 等待 iframe 初始化完成。
  • 往 iframe 里 postMessage。
  • 再定时去问 iframe：“结果出来了吗？”

这就是本地运行时路线的感觉。它能跑，但每个 iframe 都要占资源，初始化也复杂。

ZephyrSail：直接调接口并消费流

它更像正常前端应用：

  • 准备请求 body。
  • 带上 token 和 session 信息。
  • 直接 fetch 官方接口。
  • 边读 stream 边渲染文字。

难点是 body 不是 JSON，而是 protobuf；所以它用模板文件当“请求骨架”，再把动态字段填进去。

Current Engine

旧路线的调用链

旧路线的核心问题是：后端先启动本地 Windsurf 运行时，然后再通过本地协议跟它通信。这个方向已经不作为当前主线。

客户端请求Claude Code / sub2api 进来，请求 /v1/messages
Rust 后端选账号、读取设置、找到本地二进制路径
启动本地进程起一个常驻运行时
本地协议把用户消息交给本地运行时处理
轮询结果反复读取状态，再转成 SSE

前端视角看它的问题

  • 启动成本高：每个本地运行时都像一个常驻隐藏窗口，占内存和进程资源。
  • 部署麻烦：用户必须下载对应平台的二进制，还要配置路径。
  • 流式不够自然：轮询再转发不如直接消费远端 stream 自然。
  • 容量难控：4h4g 机器不能无限开本地进程，要做进程池、排队、回收、隔离。
ZephyrSail

ZephyrSail 的调用链

ZephyrSail 不启动本地运行时。它把 Windsurf 官方接口需要的 protobuf 请求模板放在安装包里，请求来时 patch 模板字段，然后直接请求远端 ApiServerService。

客户端请求OpenAI / Anthropic 兼容请求进来
会话定位从 prompt_cache_key、Claude Code session、fingerprint 找会话
模板 patch把 JWT、模型、消息、工具、conversationId 填进 protobuf
远端接口请求 GetChatMessage 或 ExternalChat
边读边转解析 connect frame，实时输出 SSE

它保留了哪些状态

它不是无状态转发。它把这些东西存在自己的 SessionStore 里，类似前端应用里的 store：

sessionUuid conversationId trajectoryRunId stepNumber boundApiKey adaptive token

这些状态会被写回下一次 GetChatMessage 的 protobuf body，等于把“多轮对话上下文”从 LS 进程里搬到了本地数据库/内存状态里。

Mental Model

把 ZephyrSail 拆成前端能懂的模块

1

模板就是“初始表单数据”

GetChatMessage_req.bin 像一个已经填过一遍的复杂表单。ZephyrSail 不从零创建所有字段，而是读取这个表单，把需要变化的字段替换掉。

2

patch 就是“把 store 写进 payload”

当前账号的 JWT、模型 ID、消息列表、工具定义、采样参数、会话 ID，都会被写进 protobuf 字段。前端可以理解成提交表单前的 normalizeFormData()。

3

SessionStore 就是“会话级 Zustand/Redux”

它根据 prompt_cache_key、x-claude-code-session-id、消息 fingerprint 等找到同一个会话。这样下一轮对话能复用同一组 conversationId 和 trajectoryRunId。

4

frame parser 就是“ReadableStream 解码器”

远端不是一整个 JSON 返回，而是一帧一帧返回 protobuf。ZephyrSail 收到一段 buffer，就尝试拆 frame，拆出来立刻转成文本、reasoning、tool call 或 usage。

5

兼容层就是“把一种组件 props 转成另一种 props”

OpenAI 请求、Anthropic 请求、OpenCode 请求，字段长得不一样。ZephyrSail 先转成自己内部消息结构，再统一生成 Windsurf 上游 payload。

Recommendation

现在我们怎么走

删除本地运行时路线

不再启动本地进程，不再维护进程池，也不再配置本地二进制路径。

RemoteApiEngine 作为主线

服务端读取模板、patch 请求、直接请求远端，并把上游 frame 转成下游 SSE。

先保证 Anthropic 闭环

当前优先覆盖 /v1/messages 流式和非流式，再补工具、图片、reasoning 和 OpenAI 兼容。

建议落地顺序

  1. 读取模板目录，支持 gzip / connect frame 解包。
  2. 把 GetUserStatus 跑通，拿真实账号状态、模型、配额。
  3. 实现 SessionStore：会话 key、conversationId、trajectoryRunId。
  4. 实现 GetChatMessage 请求构造和流式 frame parser。
  5. 把 Anthropic /v1/messages 接到 RemoteApiEngine。
  6. 再补工具调用、reasoning、usage、Adaptive、ExternalChat。

注意：远端模板方案不是“简单复制 ZephyrSail”。我们需要自己理解字段和错误处理，尤其是证书、代理、上游 IP 解析、敏感信息日志脱敏、模板更新机制。这些不能盲目照搬。
