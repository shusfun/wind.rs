项目定位

这是一个新项目，不是给旧项目做兼容补丁。目标是以 Rust 后端 + React 前端 重构 WindsurfAPI 的核心能力，让 sub2api 后续只通过 127.0.0.1 调本项目提供的上游接口。

关键原则：不改 sub2api；不引入 Node sidecar；协议层不要闭门自研，必须参考 dwgx/WindsurfAPI 原版；前端 UX 尽量复刻原版管理台，但实现使用当前 React/Tailwind v4 工程。

长期方向

  • 后端用 Rust 承担账号池、并发控制、粘性会话、限流、真实 Windsurf LS 调用、兼容 OpenAI/Anthropic API。
  • 前端用 React 管理账号、代理、批量登录、调用日志、容量参数、账号测试。
  • 生产环境由 Rust 托管构建后的前端静态文件；开发环境前端单独跑 Vite 热更新。

WindsurfEngine 当前状态

当前核心文件是 crates/server/src/engine/mod.rs 和 crates/server/src/engine/proto.rs。这不是最终完成版，但已经不是占位输出。

已实现

  • 从设置读取 lsBinaryPath、lsDataDir、lsMaxInstances、轮询/超时参数。
  • 使用 tokio::process::Command 启动 Windsurf language server。
  • 使用 h2 直连本地 LS，手工 gRPC frame。
  • 实现 InitializeCascadePanelState、AddTrackedWorkspace、UpdateWorkspaceTrust、Heartbeat warmup。
  • 实现 StartCascade、SendUserCascadeMessage、GetCascadeTrajectorySteps、GetCascadeTrajectory。
  • LS 池按账号 key + 代理隔离，避免多账号共享同一 LS session。
  • /admin/account-test 和 /v1/messages 已改走共享 WindsurfEngine。

仍需补齐

  • 当前只做 Cascade 基础路径；Raw legacy 未做。
  • 图片输入未做。
  • native tool bridge 未做。
  • token usage 还不是原版完整逻辑。
  • 当前 cascade_stream 内部仍偏批量收集后 yield，后续要改成边 poll 边 yield。
  • settings 里的 lsBinaryPath 当前随后端启动加载，改设置后需要重启后端。
  • 还没有跑真实 Windsurf 官方请求验证；这是外部生产 API 调用，需要操作者确认。

关键实现判断

  • 不要把 LS Engine 做成 Node sidecar。当前方向是 Rust 后端直接管理 LS 进程与 gRPC。
  • 不要重新发明协议。字段号、调用顺序、stall/retry 策略都以原版 WindsurfAPI 为真相源。
  • 4h4g 服务器上不能为每个请求无限起 LS。要靠账号池、并发控制、LS 池上限、粘性会话、队列和限流共同控制。

必须下载与本地配置

Windsurf language server 二进制不要默认随仓库提交。需要在文档/README 提示使用者下载对应平台版本，然后在管理台 settings 或数据库里配置 lsBinaryPath。

本地已见到的二进制

  • WindsurfAPI/.local/bin/language_server_macos_x64

注意本机如果是 Apple Silicon，macos_x64 可能需要 Rosetta 或换 arm64 版本。服务器通常需要 Linux x64 版本。

后端相关设置

设置项说明
lsBinaryPathLS 二进制绝对或相对路径。未配置时真实调用会返回“请先配置 LS 二进制路径”。
lsDataDirLS 数据目录，默认使用后端数据目录下的 ls。
lsMaxInstancesLS 进程池上限，默认 20。4h4g 机器必须谨慎设置。
cascadePollIntervalMsCascade 轮询间隔，默认 500ms。
cascadeMaxWaitMs单次 Cascade 最大等待，默认 600000ms。
cascadeWarmStallMs有输出后长时间无增长的接受/停止阈值，默认 45000ms。
